If you would like to contribute to this project, please open an issue or a pull request. We are always looking for ways to improve the project and would love to hear your ideas.

The latest changes are always in master branch, so please make your Pull Request against that branch.

## âœ‰ï¸ Git Message Format

This repo adheres to the [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) specification.
Commit messages are enforced through [commitizen](https://github.com/commitizen-tools/commitizen) and a [pre-commit](https://pre-commit.com), please use `make hooks` to install them.

This leads to more readable messages that are easy to follow when looking through the project history and also allows to generate changelogs automatically.

### â†’ Allowed prefixes for commit messages

- **feat** â€” A new feature
- **fix** â€” A bug fix
- **perf** â€” A code change that improves performance
- **docs** â€” Documentation only changes
- **style** â€” Changes that do not affect the meaning of the code (white-space, formatting, missing semi-colons)
- **deps** â€” Changes to the package.json or package-lock.json
- **refactor** â€” A code change that neither fixes a bug nor adds a feature
- **ci** â€” Changes to our CI configuration files and scripts
- **test** â€” Adding missing tests or correcting existing tests
- **revert** â€” Reverting a previous commit
- **build** â€” Changes that affect the build system or external dependencies (example scopes: pnpm, renovate, github actions)
- **chore** â€” Other changes that don't modify src or test files
- **security** â€” A code change that fixes a security issue
